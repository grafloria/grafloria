Domain API architecture — how should developers build & edit typed diagrams?

The question you asked: “a class UML inherits from node and adds the extra features so I can use it cleanly — same for tables and fields. Is that what we have?” Answer: no — today it is data + functions. This page lays out the facts, what the industry converged on, four candidate architectures with the same task written in each, a weighted scorer you can tune, and my recommendation.

1 · Facts on the ground (verified in the codebase today)

KIT API
Building is declarative (erDiagram({entities}) → spec → render()); editing is free functions over the live instance: updateEntity(api, id, delta), addColumnAt(api, id, col, i), bindRowInteractions(api). Undo, port reconciliation and card re-render all live behind updateEntity.
MODEL
Every node is a plain NodeModel. “Table-ness” is data: metadata.kitEntity (the spec), metadata.html (the card), row-pinned ports. No X extends NodeModel exists anywhere in the engine.
THE CONSTRAINT
DiagramModel.fromJSON hardcodes NodeModel.fromJSON(data) (DiagramModel.ts:651) — there is no type→class registry. Nodes are reconstructed from data on undo-restore, paste/duplicate, collab op application, serialization and sync. Any subclass instance would silently degrade to a plain NodeModel at each of those points.
PRECEDENT
The engine’s native idiom is registries + type strings: the layout registry (engine.layout(name)), the element-level registerNodeType(type, renderer) rendering registry, port data-type registry — and dormant domain vocabularies already sit in engine/types/domain/{erd,uml,bpmn,flowchart}.ts ('erd:entity', 'erd:table'… wired for a validation TypeRegistry, currently consumed by nothing).
SUBCLASS PRECEDENT
StrokeModel earned true subclass status — but as a sibling entity kind (own map, own ops, own render path), not a flavor of node. A UML class is still a rectangle node with an HTML card.

2 · What the industry converged on

React Flow

// type string + data + component map
{ id:'t1', type:'erTable', data:{ columns:[…] } }
<ReactFlow nodeTypes={{ erTable: ErTableNode }} />
Pure data records. Zero entity classes for users. Editing = immutable setNodes updates.

tldraw

// records + a per-type BEHAVIOR class (ShapeUtil)
class CardShapeUtil extends ShapeUtil<CardShape> {
  static type = 'card'
  getGeometry(shape) {…}  component(shape) {…}
}
The modern answer: state stays a serializable record; behavior lives in a registered class per type. Inheritance exists — on the util, never the record.

JointJS

// real element subclasses + a NAMESPACE registry
joint.shapes.standard.Rectangle  // dia.Element subclass
// fromJSON works because type:'standard.Rectangle'
// is looked up in the joint.shapes namespace
The one subclass-based mainstream lib — and it pays with a global constructor registry threaded through deserialization. Proof it CAN work, and proof of the tax.

GoJS / X6

// GoJS: category → template, data untouched
diagram.nodeTemplateMap.add('table', tableTemplate)
// X6: config registration (optional subclass sugar)
Graph.registerNode('er-rect', { … })
Registration + data both times. X6 offers Node.define() sugar that compiles down to the registry.

n8n itself

// nodes are DESCRIPTION OBJECTS in a registry
class HttpRequest implements INodeType {
  description = { name, inputs, outputs, properties }
}
Classes exist but hold only description + execute() — the workflow graph itself stores plain JSON nodes.

Takeaway

Nobody ships user-facing entity subclassing without a deserialization registry (JointJS), and the modern libraries (tldraw, RF, X6, GoJS, n8n) all converged on data records + registered per-type behavior. Clean “OO feel” is delivered by handles/utils, not by subclassing the stored entity.

3 · Four candidate architectures — the same task in each

The task, identical in every tab: create a table · rename it · add a column at index 2 · rename a field · react to row selection · undo once.

effort: Srisk: noneDX: functional, verbose
const api = render(erDiagram({ entities:[{ id:'ORDERS', columns }] , editable:true }), host);

updateEntity(api, 'ORDERS', { name: 'Sales Orders' });
addColumnAt(api, 'ORDERS', { name:'status', type:'varchar' }, 2);
renameColumnAt(api, 'ORDERS', 3, 'cust_id');
host.addEventListener('axk:row-select', e => panel.show(e.detail.selected));
api.getEngine().undo();

Pros

  • Already shipped, tested, mutation-proven (undo, port glue, collab-safe)
  • Tree-shakeable, zero object lifetime questions
  • Matches React Flow’s mental model — familiar

Cons

  • Repetitive (api, id, …) threading; stringly-typed ids at every call
  • No discoverability — you must know the function names
  • Domain concepts (field, column) have no type identity — “not clean” is a fair critique
effort: S–Mrisk: none (stateless veneer)DX: the OO feel you asked for
const t = erTable(api, 'ORDERS');          // ErTable extends CardHandle — inheritance, but SAFE:
t.name = 'Sales Orders';                   // a handle is (api, id) — it stores NOTHING
t.columns.add({ name:'status', type:'varchar' }, { at: 2 });
t.columns.get('customer_id').rename('cust_id');   // ErField handle; ports stay glued
t.onRowSelect(sel => panel.show(sel));
t.undo();                                   // sugar for engine.undo()

const c = umlClass(api, 'Shape');          // UmlClass extends CardHandle
c.abstract = true;  c.methods.add('+ area(): float');

Pros

  • Exactly the clean usage you described — with a real class hierarchy (CardHandle → ErTable/UmlClass, ErField) where it is safe: the façade
  • Handles are stateless → they can never go stale across re-render, undo, collab, reload; nothing to serialize
  • Every mutation still funnels through updateEntity → one-step undo + port reconciliation come free
  • Full TypeScript autocomplete on domain concepts; ids typed once
  • Small: a typed veneer over update.ts / rows.ts / editing.ts

Cons

  • Two API layers to document (spec-in, handles-after) — needs a clear “build with data, edit with handles” story
  • Doesn’t by itself let THIRD PARTIES define new card kinds (that’s option C)
  • Property setters that mutate (t.name = …) surprise some FP-minded users — offer methods too
effort: M–Lrisk: moderate (new public contract)the platform play
// tldraw-ShapeUtil-style: data records + a registered KIND with behavior + a generated handle
defineCardKind({
  kind: 'er-table',                        // stored as node metadata.kind — survives serialization
  build(spec)      {…},                     // spec → html tree + size + ports (er.ts's entityCard)
  reconcile(delta) {…},                     // port glue on edit (update.ts's matcher)
  chrome: { editable:{…}, rowSelect:true }, // finalize-time bindings
  handle: ErTable,                          // option B's class, now provided BY the kind
});

const t = api.card('ORDERS');              // registry resolves kind → ErTable, fully typed
t.columns.add({ name:'status' }, { at:2 });

// and now a USER can do what WE did for n8n:
defineCardKind({ kind:'n8n-node', build: n8nCard, handle: N8nNode });

Pros

  • Extensibility: third parties define their own typed cards (n8n nodes, BPMN, org charts) with the same power the ER/UML kit has — “the more reusable, the more they love us”
  • Includes option B (handles come from the kind) — B is a strict subset
  • Kind string lives in metadata → serialization/collab-safe by construction; the dormant erd:*/uml:* vocabularies finally get a consumer
  • Matches the engine’s registry idiom (layout registry, node-type renderer registry)

Cons

  • A new public contract to design well ONCE (build/reconcile/chrome lifecycle) — the cost is in getting this API right, not the code volume
  • Port-reconciliation as a user-implementable hook needs guard rails (our matcher took real care)
  • Overkill if ER + UML stay the only kinds
effort: Lrisk: high — engine-invasiveDX: good until it bites
class UmlClassNode extends NodeModel {
  attributes: string[] = [];
  addMethod(sig: string) { … }
}
// …and for this to SURVIVE, every reconstruction site must know the class:
// fromJSON, undo restore, paste/duplicate clone, collab op apply, sync
NodeClassRegistry.register('uml-class', UmlClassNode)   // ← new engine seam, everywhere
The trap, concretely: today DiagramModel.fromJSON builds NodeModel unconditionally. Without threading a class registry through every reconstruction path, a UmlClassNode silently becomes a plain NodeModel on the first undo, paste, or collab round-trip — methods gone, fields gone, no error. With the registry, every one of those safety-critical paths (undo, clipboard, collab, sync, serialization) takes a permanent maintenance tax. JointJS proves it’s possible; tldraw/RF/X6/GoJS/n8n all chose not to.

Pros

  • The most literal match to “UML class inherits from node”
  • Instance methods sit directly on the thing you drag

Cons

  • Touches the engine’s most safety-critical seams for an ergonomic gain B already delivers
  • Call-sites end up looking identical to B’s handles anyway
  • Collab whole-card LWW + class state = two sources of truth to keep converged

5 · Recommendation

Ship B now, design C as its second step, never do D.
  • B (façade handles) is the clean usage you asked for — a real CardHandle → ErTable / UmlClass hierarchy with ErField handles — at small cost and zero architectural risk, because handles are stateless and every write funnels through the already-proven updateEntity (undo + port glue included). It can ship quickly and nothing about it is throwaway.
  • C (card-kind registry) is where B’s classes come from once third parties should be able to define their own typed cards (the n8n demo hand-rolled exactly what a kind would package: card builder + ports + chrome). B’s handle classes become the kind’s handle — B is a strict subset of C, so nothing is wasted. Decide C when a second consumer beyond ER/UML is real.
  • D buys the same call-site ergonomics as B at engine-invasive cost, and the industry consensus (tldraw/RF/GoJS/X6/n8n) is data + registered behavior — the engine’s own registries and dormant erd:*/uml:* vocabularies point the same way.
