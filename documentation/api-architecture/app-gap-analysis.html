Can we build Figma / draw.io / Visio / a dashboard builder on Grafloria?

A gap analysis of the library, not of app features: what Grafloria would need so these apps are easy to build on top. Four parallel research passes over the real sources; every row below is backed by a file:line or a live probe. Verdicts are deliberately blunt.

1 · The answers, in one screen

📊 Dashboard builder

widgets · stacked/flow · charts inside
A quarter
Viable. Nothing architecturally blocked.
Charts in widgets already work — via custom nodes, not the HTML sanitizer. A 12-column dashboard layout engine is already written and never invoked.

📐 draw.io / Visio

stencils · connectors · pages · layers
2–4 dev-months
To a credible draw.io-lite.
An outstanding diagramming canvas; not yet a diagramming document engine. ~60% of the work is exposure, not construction.

🎨 Figma-class

vector · text · components · frames
Substrate, not foundation
4 large subsystems + 5 medium + ~10 small.
You'd get canvas, collab, export, theming free (12–18 months of the hardest infra) then write the entire design model yourself.

2 · The finding that matters more than any single gap

Grafloria's biggest weakness is not missing capability — it is unreachable capability. All four research passes independently hit the same shape: subsystems that are built, tested, and wired to nothing. This is the single highest-leverage investment area, because the code already exists.

Top blockers

  1. Export of widget content (L). PDF drops foreignObject by design; PNG rasterizes a serialized SVG via drawImage, which won't paint HTML — and custom-node content isn't in the vnode tree at all. Worse, svg-renderer.ts:1764 discards the fidelity warnings, so you'd export a dashboard and get blank rectangles with no diagnostic. For a BI product this is table stakes and needs a genuinely new subsystem.
  2. Custom nodes are never culled (M). Probed live: hosts are created for every node regardless of viewport and never removed — panned 50,000px away, charts still ticking. 100 widgets = 100 live chart instances forever.
  3. Container resize doesn't reflow children (M). setSize() doesn't propagate, setFrame() never calls applyLayout. Resize a container or delete a widget and everything else stays put.
  4. Flex/grid item config is dead metadata (S–M). Verified above. You cannot place a widget in an explicit grid cell.
  5. No node z-order; per-node lock is UI-only (S). Groups have bringToFront; nodes have nothing. state.locked is honoured by input layers but by no model mutator — auto-layout relocates locked nodes.

Top blockers

  1. There is no document model — only a diagram model. No layers, no pages; DocumentEnvelope holds exactly one diagram. The only architectural blocker here: it changes serialization, the envelope, the CRDT surface and every export path.
  2. The stencil system is built and orphaned. A palette is the #1 thing this app class needs and all the parts are on the floor.

Top blockers

  1. No vector path entity, and no boolean ops. That is the product category. Cruel irony: canvas/path-geometry.ts is already a complete path kernel (M/L/C/Q/Z, parse/transform/flatten/bounds/hit-test) that no model consumes.
  2. Text is a property, not an object — no standalone text, no style runs, and wrap decisions come from length × fontSize × 0.6 while a real measureText sits unused in the canvas backend. Typography is most of design work.
  3. No component/instance system. templateId is written once and never read; edit a master, nothing propagates. Design systems are why teams buy Figma.
  4. No document structure — no frames, clipping, pages, or group transform. Deepest issue: the renderer emits flat siblings, not nested <g>, so parent transforms never reach children; multi-select rotation is hardcoded to zero.
  5. The mutation/undo seam is diagram-shaped and closed. A design tool is ~90% property edits and there is no node style command — setStyle() is a raw write outside undo. Cheapest of the five to fix; silently poisons everything built above it.

Top foundation blockers

  1. The front door is not the door. 238 capability symbols unreachable; the flagship in-repo app uses @grafloria/element in zero files. It's the multiplier on every other gap — excellent a11y, layout, collab and undo machinery all exist and are invisible to an app author. And the existing lock is an allowlist, so the gap regrows every wave.
  2. No entity-kind registry. The one gap with no workaround: fromJSON hardcodes concrete classes at 10 sites and the collab EntityTarget union is closed across 4 files. Tellingly, the library's own domain features (swimlanes, ER tables, UML, comments) are all metadata conventions working around the missing primitive.
  3. Undo is a good machine bolted to an unguarded model. ~30 of ~104 document mutators covered (29%), ~66 internal bypass sites, no style/metadata/layout command, a competing layout undo stack — and with collab attached, two unconnected Ctrl-Z implementations over one document.
  4. Process-global renderer registries. Shape/style/tool registries are module-scope Maps, so two diagrams on one page cannot have different vocabularies — an ordinary requirement (widget editor beside a live preview) hits a hard ceiling. The correct instance-scoped pattern already exists twice in-repo.
  5. Three live collab bugs, all S-effort, none covered by tests: group membership over collab corrupts a Set into a character-split string and then makes serialize() throw (document unsaveable — reproduced live); manual waypoints never propagate; writeGeneric calls a 1-arg setter into 2-arg setData, writing a key literally named [object Object].

Recommended hardening sequence

  1. Phase 0 — Stop the bleeding (days). Generate the element barrel from engine/renderer capability exports and invert the lock: fail on an unexported capability, not a missing allowlist entry. Un-orphan clipboard/ and selection/ (absent from the engine barrel entirely). Add exports/files maps; stop shipping __mocks__.
  2. Phase 1 — Ship what already exists (1–2 weeks). Highest value-per-hour in the whole plan: export the command system, core models, ExtensionHost, LayoutRegistry, the serialization suite and the a11y controllers; wire a11y + keyboard nav into createDiagram() (Angular is a working reference). Delivers a11y, keyboard, layout and undo to every non-Angular consumer without writing a feature.
  3. Phase 2 — Fix correctness (~1 week). The three collab bugs, the un-undoable delete-link path, label-command merging (today: one undo entry per keystroke), palm rejection, ephemeral state out of saved docs.
  4. Phase 3 — Registry coherence (3–4 weeks). Instance-scope the renderer registries; unify ~34 registries across ~6 idioms onto one contract; make ExtensionHost the single entry point. Decide PluginManager's fate — install()/activate() never fire and zero plugins exist; a dead plugin system in the public API is worse than none.
  5. Phase 4 — defineCardKind (6–10 weeks). The entity-kind registry (Option C from the earlier decision). Last of the build work deliberately: it touches safety-critical seams, and Phases 0–3 give it a stable registry contract, trustworthy undo and correct collab to build against.
  6. Phase 5 — Undo invariant + docs. Close the bypass sites (or enforce "document write requires a command or explicit runSystemWrite()"); reconcile the two undo stacks; rewrite the root README (currently unmodified Nx boilerplate), delete libs/renderer/README.md's fiction (it documents 4 classes that no longer exist), tag @public/@internal.

4 · What to invest in first

Each initiative scored 0–10 on how much it unlocks per app class + how cheap it is. Drag the weights to match your intent; the ranking re-sorts live.

5 · What Grafloria is already unusually strong at — do not rebuild these

  • Collaboration. A hand-built per-property LWW CRDT with Lamport clocks that explicitly rejects whole-entity LWW ("if you drag a node while I rename it, somebody's work is silently lost"), 4 transports incl. serverless BroadcastChannel, fuzz + hostile-transport suites, supersession-aware undo. This is exactly the multiplayer model a design tool needs, already correct.
  • Presence. Cursors live outside the VNode tree because the obvious implementation costs 240 full re-renders/sec at 4 peers. Measured 0.0ms idle-frame delta.
  • Routing — better than the thing you'd be cloning. Seven pluggable routers plus an off-thread libavoid-class global solver, live rerouting, jump-overs, draw.io's floating attachment ported verbatim.
  • Export. True vector PDF with selectable text; per-object export that prunes the tree (treating crop-only as "an information leak"); raster backends that refuse formats they can't produce rather than emitting wrong bytes; and the exported SVG/PNG is the document (metadata + PNG iTXt + checksum reimport) — a better round-trip than draw.io's.
  • Persistence. PersistedDocument + a stepwise migration registry explicitly open for host apps, full metadata round-trip, document checksums for dirty-checking.
  • Theming. Token bridge with shadcn/MUI/Tailwind presets, fail-soft var() fallbacks, a named-style registry wired to cache invalidation — genuine edit-once-update-all shared styles.
  • Snapping geometry. Edge/centre alignment, equal-spacing detection in BETWEEN and CHAIN patterns, distance labels. The hard math for align/distribute is already written — only the command layer is missing (an S-sized win).
  • Performance + SSR. Culling, spatial index, LOD governor, off-thread layout; 10k nodes: zoom-out 63s → 102ms, drag 5.5s → 61ms. Real headless render with determinism tests that first assert there's no DOM.

6 · The honest summary

Grafloria is a world-class diagramming canvas with production-grade substrate (collab, export, theming, routing, performance) and a diagram-shaped model. That combination makes the dashboard builder a straightforward quarter, draw.io-lite a 2–4 month exposure-and-document-model project, and Figma-class a "build the design model beside it" undertaking. If the goal is apps get built on Grafloria easily, the highest-leverage work is not new features — it is closing the reachability gap (export the commands, own the template registry, wire the membership services, expose in-place editing) and then adding the document model (layers + pages) that two of the three app classes both need.
